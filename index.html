What is technical debt?

Strategic decisions to take shortcuts for short term gain or choosing a short-sighted solution instead of investing in long term benefits. Consequence of poor design and architecture.

This term often gets conflated with poor coding practices or legacy code - but it's bigger than that! Technical debt is STRATEGIC.

Like financial debt, debt that is not paid down will grow compounding interest over time.

*NOTE* Not all technical debt is bad. Some, like prototypes, are neccessary for project advancement.

This is often seen in practice as:

  • Old versions of WordPress and plugins
  • Dated technology or infrastructure
  • Overly complicated code
  • Insecure code
  • Lack of validation
  • No documentation
  • Lack of or incomplete coding standards

Imagine your team has...

  • Timeline bottlenecks due to lack of resources
  • Codebases that increase task completion speed
  • Projects over budget
  • Fragile applications that require heroics
  • Deployments causing sites to fall over sideways

So, how do we tackle this?

Capacity and demand

Identify the capacity

Before you can repay any technical debt, you need a clear understanding of the capacity, including both work in progress (WIP) and active commitments.

Account for all the work you do:

  • Business projects - client projects or organizational priorities that drive revenue.
  • Internal projects - technical department projects, these can be investment or tied to other teams' needs.
  • Deployments/change management - planning, testing and ensuring that multiple releases don't conflict.
  • Unplanned work - outages, critical bugs and other issues that prevent actual work time.

Technical debt's interest comes in unplanned work

Start measuring how much time is lost to unplanned work.

Align priorities

  • Have a clear understanding of the end to end business priorities and how projects fit in.
  • Have conversations with stakeholders and business leaders on their expectations and assumptions for projects.
  • Projects you invest the most in should also have the highest business priority.

Make an inventory

  • Identify the project, the resource(s) working on it and deadline
  • Don't limit this to only client billable work
  • Map each project to its work type and business priority (high/medium/low)
  • Look for patterns and conflicts

Get organized, plan ahead

  • Kanban board
  • Gant chart
  • Shared calendar

Don't Jeopardize Commitments

Before accepting new work look at the existing WIP to determine if this will jeopardize existing commitments.

Why this matters...

  • Without capacity analysis before accepting new work, developers will be scrambling and electing to take shortcuts to meet deadlines.
  • This means fragile applications in production, which creates more unplanned work and increases technical debt.
  • Quality and due date performance go up by allowing developers to focus on a set number of projects.

Make time for priority debt

  • You will never eleminate all technical debt.
  • Ensure the savings of the debt justifies the cost to pay it down.
  • Prioritize your debt by whether or not it supports business objectives.

Identify and control the contstraints

What is a constraint?

A chain management term for something that limits a person, or an organization, in achieving a goal. These can cause bottlenecks in workflow and capacity.

Why this matters...

  • Can only move as quickly as constraints will let you.
  • Constraints contribute to capacity overage.
  • A factor in creating technical debt.

Constraints can look like...

  • Role limitations - ex. 1 person who can do a task
  • Infrastructure limitations - ex. legacy systems
  • Time limitations - ex. shortend timeline
  • Process limitations - ex. unclear release cycles

After identifying...

  • Make quick improvements to get work through faster.
  • Make sure your process accounts for its limitations.
  • Consider what else you can do to eliminate it.

Constraint example:

FRAGILE INFRASTRUCTURE MANAGED BY ONE PERSON

  • Document the process for reviving servers.
  • Train up team members on the process for flexibility in addressing urgent needs.
  • Upgrade infrastructure or move to a managed solution.

What's the impact?

  • Improved capacity
  • Reduced task lead time
  • Increased project completion speed
  • Increased profit

Standardize

Without standards, it's like the wild west.

A few things to avoid

Handling special cases 

						
							if ($status == "awake"){ 
								$activity = "Writing spaghetti code"; 
							} elseif($healthstatus == "OK"){ 
								$activity = "More spaghetti code"; 
							}else{ 
								print "Refactor this code"; 
							}

A few things to avoid

Use of Important in CSS/SASS/LESS 

						
							header { 
    						 color: #e74c3c; 
							}
							header.intro {
    						 color: #3498db !important;
    						 font-size: 20px;
							}

A few things to avoid

No or useless comments 

						
							# Monkeys and bananas

							foreach(banana b in bunch) {
    							monkey.eat(b);  
							}

Standardization means:

  • Version control... you better be using it!
  • Clear release cycles or continuous integration
  • Coding standards and code review
  • Commenting and code documentation
  • Process documentation of common tasks
  • Testing

Why this matters...

Standardization is a key to controling your constraints, making tasks repeatable so that others can execute and enforcing a level of consistancy and quality in the product.

Release Cycle Management

  • "Done" often means ready for QA, but usually there are still bugs.
  • Using gitFlow with a master branch that is always ready to ship and automating regression testing can improve the quality being deployed.
  • Consult your commitments when scheduling releases - be wary of two high-risk deployments in a close window.

Code Standards

  • Document methods and measures for coding to create a shared understanding, more efficency and organization.
  • Baseline using industry standard features, frameworks and technologies.
  • Take inspiration from WordPress' Theme and Plugin Developer Handbooks
  • Make security and compliance part of your process. They are only effective if they are applied before your project ships.

Code Documentation

  • Within your code, document what it does and why.
  • Documentation only makes sense if it's meaningful AND gets updated when the code evolves.
  • Build documentation into your code standards and code review process.
  • Documentation does NOT mean commenting out legacy code instead of refactoring.

Process Documentation

  • Process documentation and abstraction allows tasks to scale across people and groups.
  • Take recent requests, document what the steps are, who can do them and how long they take.
  • Will cut down on onboarding time for new hires.

Testing

  • QA testing is a critical component to preventing future technical debt.
  • Manual testing will only get you so far. Automated testing can catch signifcant bugs that could be missed.
  • Tests look at the functionality, not the implementation. They should always be done in conjunction with a code review.

Standardize Mainenance

Building preventative maintenance into your workflow helps ensure availability and security. The WordPress community works hard to put out new releases - it's your responsibility to apply them!

To sum it up...

Identify your top areas of technical debt to tackle to decrease the amount of unplanned work created by problematic code, inefficient workflows and disorganization. Always be evaluating your process and what can be improved.
