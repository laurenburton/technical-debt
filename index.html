What is technical debt?

Strategic decisions to take shortcuts for short term gain or choosing a short-sighted solution instead of investing in long term benefits.

It's more than bad code! Technical debt is STRATEGIC.

Like financial debt, debt that is not paid down will grow compounding interest over time.

*NOTE* Not all technical debt is bad.

This is often seen in practice as:

  • Old versions of WordPress and plugins
  • Dated technology or infrastructure
  • Overly complicated code
  • Insecure code
  • Lack of validation
  • No documentation
  • Lack of or incomplete coding standards

Capacity and demand

Identify the capacity

You need a clear understanding of the capacity, including both work in progress (WIP) and active commitments.

Account for all the work you do:

  • Business projects
  • Internal projects
  • Deployments/change management
  • Unplanned work

Technical debt's interest comes in unplanned work

Align priorities

  • Have conversations with stakeholders
  • Invest in projects with high business priority

Make an inventory

Identify...

  • Project
  • Resource(s)
  • Business priority
  • Deadline
  • Look for patterns and conflicts

Get organized, plan ahead

  • Kanban board
  • Gant chart
  • Shared calendar

Don't Jeopardize Commitments

Look at the existing WIP before saying yes

Why this matters...

  • Less developers scrambling
  • Less shortcuts
  • Less fragile applications
  • Higher quality and due date performance

Make time for priority debt

  • You will never eleminate it all
  • Justify the cost to pay it
  • Prioritize your debt

Identify and control the contstraints

What is a constraint?

A chain management term for something that limits a person, or an organization, in achieving a goal.

Constraints can look like...

  • Role limitations
  • Infrastructure limitations
  • Time limitations
  • Process limitations

Constraint example:

FRAGILE INFRASTRUCTURE MANAGED BY ONE PERSON

  • Unpredictable crashes
  • Bottlenecks other projects
  • Limited support coverage

Why this matters...

  • Impacts work pace
  • Contributes to capacity overage
  • Creating technical debt

After identifying...

  • Make quick improvements
  • Account for limitations
  • How can you eliminate it?

Constraint solution example:

FRAGILE INFRASTRUCTURE MANAGED BY ONE PERSON

  • Document the process
  • Train up team members
  • Upgrade infrastructure

What's the impact?

  • Improved capacity
  • Reduced task lead time
  • Increased project speed

Standardize

Without standards, it's the wild west.

A few things to avoid

Handling special cases 

						
							if ($status == "awake"){ 
								$activity = "Writing spaghetti code"; 
							} elseif($healthstatus == "OK"){ 
								$activity = "More spaghetti code"; 
							}else{ 
								print "Refactor this code"; 
							}

A few things to avoid

Use of Important in CSS/SASS/LESS 

						
							header { 
    						 color: #e74c3c; 
							}
							header.intro {
    						 color: #3498db !important;
    						 font-size: 20px;
							}

A few things to avoid

No or useless comments 

						
							# Monkeys and bananas

							foreach(banana b in bunch) {
    							monkey.eat(b);  
							}

Standardize so you dont end up with monkeys and bananas

Standardization means:

  • Version control... you better be using it!
  • Clear release cycles or continuous integration
  • Coding standards and code review
  • Commenting and code documentation
  • Process documentation of common tasks
  • Testing

Why this matters...

  • Control your constraints
  • Replicate tasks
  • Consistancy and quality

Release Cycle Management

  • Still bugs after it's "done"
  • Use gitFlow - always ready to ship
  • Plan releases in advance

Code Standards

  • Document methods
  • Baseline on industry standard features
  • Use WordPress' Theme and Plugin Developer Handbooks
  • Plan for security and compliance

Code Documentation

  • Document what it does and why
  • Keep it updated!
  • Include in code review
  • Does NOT mean commenting out legacy code

Process Documentation

  • Scale work across people
  • Document recent requests
  • Cut down on onboarding time for new hires.

Testing

  • QA testing is critical
  • Incorporate automated testing
  • Tests look at the functionality, not the implementation

Standardize Mainenance

Building preventative maintenance into your workflow helps ensure availability and security.

To sum it up...

  • Identify top debt
  • Decrease unplanned work
  • Evaluate for continuous improvement
